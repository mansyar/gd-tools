# Code Audit Report — gd-tools

**Date:** 2026-07-13
**Auditor:** Software Auditor (automated)
**Scope:** Full codebase review of `gd-tools-cli` v0.1.0
**Methodology:** Static analysis, source code review, runtime verification

---

## Executive Summary

This audit reviewed the full `gd-tools` codebase — a Python CLI for GDScript development workflows (test, lint, format, coverage) targeting Godot 4.5+. The codebase is well-structured with clear separation of concerns, comprehensive Pydantic v2 configuration models, and extensive test coverage. However, the audit identified **23 findings** across five severity levels:

| Severity   | Count | Description                                        |
| ---------- | ----- | -------------------------------------------------- |
| CRITICAL   | 5     | Bugs causing silent failures, crashes, or data corruption |
| HIGH       | 5     | Functional defects or risky patterns with real impact     |
| MEDIUM     | 7     | Correctness issues with limited blast radius              |
| LOW        | 6     | Code quality, dead code, minor inconsistencies           |

The most impactful issues cluster around two themes:
1. **CLI flags silently ignored** — `--timeout` and `--min` are accepted but not forwarded to the test runner in certain code paths, violating the principle of least surprise.
2. **Coverage pipeline fragility** — the coverage subsystem has error-masking, assertion-based control flow, and format mismatches that can crash at runtime.

## Findings

### CRITICAL

#### C1. `--timeout` flag ignored without `--coverage`

**Location:** `src/gd_tools/cli.py` (test command, `--coverage` else branch)
**Related:** `src/gd_tools/test_runner.py` — `run_tests()` accepts `timeout` parameter

**Description:**
The `test` command's `--timeout` CLI option is correctly passed to `run_coverage_test()` when `--coverage` is enabled, but it is **not forwarded** to `run_tests()` in the non-coverage branch. The non-coverage path calls `run_tests()` without the `timeout` argument, causing it to default to `300` seconds regardless of what the user specified.

**Impact:**
```
gd-tools test --timeout 60    # silently uses 300s timeout
```
Users relying on `--timeout` for non-coverage test runs will see no effect. In CI environments where a fast-fail timeout is critical, this can cause pipelines to hang for 5 minutes instead of failing at 60 seconds.

**Recommended Remediation:**
Pass `timeout=timeout` to the `run_tests()` call in the non-coverage else branch.

**Verification:**
- Add a unit test that mocks `run_tests` and asserts `timeout` is forwarded.
- Add an integration test: `gd-tools test --timeout 1` with a long-running test suite should timeout at ~1s.

#### C2. "text" vs "terminal" coverage format mismatch

**Location:**
- `src/gd_tools/config.py` — `CoverageConfig.format` validator allows `("html", "lcov", "cobertura", "text")`
- `src/gd_tools/coverage/reporter.py` — `_SUPPORTED_FORMATS = {"html", "lcov", "cobertura", "terminal"}`

**Description:**
The Pydantic config model validates `format="text"` as valid, but the reporter's supported formats set uses `"terminal"` instead of `"text"`. A user who sets `format = "text"` in `gd-tools.toml` (or passes `--format text`) will pass validation but crash at report generation with a format-not-supported error.

The README and user documentation also reference `"text"`, creating a three-way inconsistency: docs say `"text"`, config allows `"text"`, but the reporter expects `"terminal"`.

**Impact:**
Any user attempting to use the terminal/text coverage format via configuration will hit a runtime crash. This is a config-validates-but-runtime-fails defect.

**Recommended Remediation:**
Align all three to a single canonical name. Recommendation: use `"text"` everywhere (matches the conventional name in other coverage tools like `coverage.py`), or accept both as aliases in the reporter.

**Verification:**
- Unit test: `format="text"` in config produces a terminal report without error.
- Unit test: `format="terminal"` also works (if aliasing is chosen).

#### C3. `_format_rate` returns invalid Cobertura rate strings

**Location:** `src/gd_tools/coverage/cobertura_reporter.py` — `_format_rate()` function

**Description:**
The `_format_rate()` function formats coverage rates as follows:
```python
f"{hit/total:.4f}".rstrip("0").rstrip(".")
```
For 0% coverage (`hit=0`), this produces `"0"` instead of `"0.0"`.
For 100% coverage (`hit=total`), this produces `"1"` instead of `"1.0"`.

The Cobertura XML schema expects rate attributes as decimal values (e.g., `"0.0000"`, `"1.0000"`). Values like `"0"` and `"1"` may be rejected or misinterpreted by Cobertura-consuming tools (Jenkins, SonarQube, etc.).

**Impact:**
Cobertura reports generated by gd-tools may fail to parse in CI/CD tools that consume Cobertura XML, or may display incorrect coverage percentages.

**Recommended Remediation:**
```python
def _format_rate(hit: int, total: int) -> str:
    if total == 0:
        return "0.0"
    return f"{hit / total:.4f}"
```

**Verification:**
- Unit test: `_format_rate(0, 10)` returns `"0.0000"`.
- Unit test: `_format_rate(10, 10)` returns `"1.0000"`.
- Validate generated XML against the Cobertura XSD.

#### C4. Test failure masked by missing coverage.json

**Location:** `src/gd_tools/coverage/orchestrator.py` — `run_coverage_test()`

**Description:**
In `run_coverage_test()`, when tests fail, the function stores the `TestFailureError` in a `test_error` variable and then proceeds to call `read_coverage_json()`. If the test failure caused GUT to crash before writing the coverage output file, `read_coverage_json()` raises a `CoveragePlanError` (file not found / invalid JSON). This `CoveragePlanError` propagates up, **masking** the `TestFailureError` that was the actual root cause.

The error precedence logic (`if test_error: raise test_error`) only executes if `read_coverage_json()` succeeds. When it fails, the user sees a confusing "coverage plan error" instead of the actionable "test failure" message.

**Impact:**
When tests fail in coverage mode, users may see a misleading error about a missing coverage file instead of the actual test failures. This makes debugging significantly harder, especially in CI where the test output (JUnit XML) may be the only clue.

**Recommended Remediation:**
Wrap the `read_coverage_json()` call in a try/except. If it fails and `test_error` is set, re-raise `test_error` (the root cause). Only raise the coverage error if tests actually passed.

```python
try:
    coverage_data = read_coverage_json(output_path)
except CoveragePlanError:
    if test_error:
        raise test_error
    raise
```

**Verification:**
- Integration test: simulate test failure + missing coverage file → assert `TestFailureError` is raised, not `CoveragePlanError`.

#### C5. `assert` used in production control flow

**Location:** `src/gd_tools/coverage/orchestrator.py` — `assert result is not None`

**Description:**
The orchestrator uses `assert result is not None` as a control-flow guard. When Python is run with the `-O` (optimize) flag, all `assert` statements are stripped from bytecode. This means the guard disappears entirely in optimized mode, and `result` (which may be `None`) proceeds to the next line, likely causing an `AttributeError` with a confusing traceback.

**Impact:**
In optimized Python environments (some CI/CD setups, Docker images using `python -O`), the coverage pipeline may crash with an unrelated `AttributeError` instead of the intended validation error.

**Recommended Remediation:**
Replace with an explicit `if` check that raises a proper exception:
```python
if result is None:
    raise CoveragePlanError("Coverage generation returned no result")
```

**Verification:**
- Run `python -O -m pytest tests/unit/test_orchestrator.py` and confirm behavior is unchanged.

#### H1. `--min` flag silently ignored without `--coverage`

**Location:** `src/gd_tools/cli.py` (test command), `src/gd_tools/test_runner.py` — `run_tests()` `min_percent` parameter

**Description:**
The `run_tests()` function accepts a `min_percent` parameter, but it is never used in the function body (a comment indicates "enforcement deferred to Phase 3"). The CLI `test` command accepts `--min` but only passes it through in coverage mode. In non-coverage mode, `--min` is silently dropped.

**Impact:**
```
gd-tools test --min 80    # no threshold enforcement occurs
```
Users expect `--min` to enforce a minimum pass rate. The silent ignore is worse than an error — it gives false confidence that the threshold was checked.

**Recommended Remediation:**
Two options:
1. **Implement:** Enforce `min_percent` in `run_tests()` by comparing against the test pass rate after parsing JUnit results, raising `CoverageThresholdError` (or a new `TestThresholdError`) if below.
2. **Reject:** If not ready to implement, raise a clear error: `--min is only supported with --coverage` or `--min is not yet implemented for non-coverage runs`.

**Verification:**
- Unit test: `run_tests(min_percent=100)` with failing tests raises an error.
- Unit test: `run_tests(min_percent=0)` with any tests passes.

#### H2. Three inconsistent `check_gut_installed` implementations

**Location:**
- `src/gd_tools/test_runner.py:159` — checks `addons/gut/gut_cmdln.gd`
- `src/gd_tools/init.py:102` — checks `addons/gut/gut.gd`
- `src/gd_tools/doctor.py:174` — checks `addons/gut/gut.gd`

**Description:**
Three separate `check_gut_installed()` functions exist, each checking for a different file to determine if GUT is installed. `test_runner.py` checks for `gut_cmdln.gd` while `init.py` and `doctor.py` check for `gut.gd`. This means:
- `init` and `doctor` may report GUT as installed while `test_runner` reports it as not installed (or vice versa) if only one of the two files exists.
- There is no single source of truth for the GUT installation check.

**Impact:**
Inconsistent GUT detection can cause confusing scenarios: `gd-tools doctor` says GUT is fine, but `gd-tools test` says GUT is not installed. This erodes user trust and makes debugging difficult.

**Recommended Remediation:**
Create a single `check_gut_installed(project_root)` function in a shared module (e.g., `godot.py` or a new `gut_utils.py`) that checks for **both** required files (or the canonical GUT entry point). All three modules should import and use this single function.

**Verification:**
- Unit test: function returns `True` when both files exist.
- Unit test: function returns `False` when either file is missing.

#### H3. Plan generator crashes on any syntax error

**Location:** `src/gd_tools/coverage/plan_generator.py` — `parse_gdscript()` and `generate_plan()`

**Description:**
`parse_gdscript()` calls `gdtoolkit.parser.parser.parse(code)` without a try/except. If any `.gd` file in the project contains a syntax error, the `LarkError` (or other parse exception) propagates up and crashes the entire coverage plan generation. There is no per-file error handling or skip mechanism.

**Impact:**
A single syntax-error file anywhere in the project (including in `addons/` if not excluded, or in WIP files) prevents coverage plan generation for the entire project. Users must fix or remove every syntax error before they can get any coverage data.

**Recommended Remediation:**
Wrap `parse_gdscript()` calls in `generate_plan()` with try/except. Log a warning for unparseable files and skip them, rather than crashing. Optionally, collect a list of skipped files and report them to the user.

```python
for gd_file in gd_files:
    try:
        code = gd_file.read_text(encoding="utf-8")
        tree = parse_gdscript(code)
        # ... process
    except Exception as e:
        skipped.append((gd_file, str(e)))
        continue
```

**Verification:**
- Unit test: `generate_plan()` with one valid and one invalid `.gd` file produces a plan for the valid file and reports the invalid one as skipped.

#### H4. `get_godot_version` has no subprocess timeout

**Location:** `src/gd_tools/godot.py` — `get_godot_version()`

**Description:**
`get_godot_version()` calls `subprocess.run([binary, "--version"], ...)` without a `timeout` parameter. If the Godot binary hangs (e.g., due to a dialog, GPU issue, or corrupted binary), this call blocks indefinitely.

**Impact:**
`gd-tools doctor`, `gd-tools init`, and any command that checks the Godot version can hang forever with no way to interrupt except a manual kill. In CI, this manifests as a stuck pipeline.

**Recommended Remediation:**
Add a `timeout` parameter (e.g., 10–30 seconds):
```python
result = subprocess.run(
    [binary, "--version"],
    capture_output=True,
    text=True,
    timeout=30,
)
```
Catch `subprocess.TimeoutExpired` and raise a `GodotNotFoundError` with a helpful message.

**Verification:**
- Unit test: mock `subprocess.run` to raise `TimeoutExpired` → assert proper error is raised.

#### H5. Init aborted by user exits with code 0

**Location:** `src/gd_tools/init.py` — user declines GUT install

**Description:**
When the user declines the GUT installation prompt during `gd-tools init`, the code calls `sys.exit(0)`. Exit code 0 conventionally means "success." In CI/CD pipelines, this means a declined init is indistinguishable from a successful init.

**Impact:**
CI scripts that run `gd-tools init` non-interactively will see exit code 0 even if init was aborted, leading to silent misconfiguration. Subsequent commands may fail in confusing ways because GUT was never installed.

**Recommended Remediation:**
Exit with a non-zero code (e.g., `1` or a dedicated code) when the user declines, or print a clear message and exit with code `2` (the `GdToolsError` exit code). Alternatively, if exit 0 is intentional for "user chose not to proceed," add a `--yes` / `--no-interaction` flag for CI and document the behavior.

**Verification:**
- Unit test: decline prompt → assert exit code != 0 (or documented behavior).

#### M1. Duplicate line entries in LCOV/Cobertura reports

**Location:**
- `src/gd_tools/coverage/lcov_reporter.py:69-81`
- `src/gd_tools/coverage/cobertura_reporter.py:95-114`

**Description:**
When an `if_branch` and `else_branch` (or multiple branches) map to the same source line (e.g., `if x: do_a() else: do_b()` on one line), the coverage reporters emit duplicate `DA:<line>` (LCOV) or `<line>` (Cobertura) records for the same line number. Some consuming tools may interpret duplicates incorrectly (double-counting) or reject the report.

**Impact:**
Coverage percentages may be miscalculated by consuming tools. LCOV's `genhtml` may emit warnings. Cobertura consumers may show inflated/deflated line counts.

**Recommended Remediation:**
Deduplicate line entries by line number before writing. For each line, aggregate hits across all branch entries that map to it.

**Verification:**
- Unit test: generate a plan with same-line branches → assert no duplicate DA records in LCOV output.

#### M2. Syntax-error files counted as "checked" in format

**Location:** `src/gd_tools/format_runner.py` — `run_format()`

**Description:**
`files_checked = len(gd_files)` is set before the formatting loop. Files that encounter `LarkError` (syntax errors) are skipped within the loop, but `files_checked` is not decremented. This means `files_checked` reports the total number of discovered files, not the number actually processed.

**Impact:**
The `FormatResult.files_checked` field is misleading. Users may believe all files were formatted when some were skipped.

**Recommended Remediation:**
Track `files_formatted` or `files_skipped` separately:
```python
files_checked = 0
for gd_file in gd_files:
    try:
        # ... format
        files_checked += 1
    except LarkError:
        files_skipped += 1
```

**Verification:**
- Unit test: one valid + one invalid `.gd` file → `files_checked == 1`, `files_skipped == 1`.

#### M3. `print()` used instead of Click/Rich for error output

**Location:** `src/gd_tools/format_runner.py` — LarkError handler

**Description:**
The `format_runner.py` uses `print(..., file=sys.stderr)` for error output, while the rest of the codebase consistently uses Click's `echo()` or Rich's `Console` for terminal output. This is an inconsistency in the output strategy.

**Impact:**
Error messages from the format command may not respect `--no-color`, `--quiet`, or other Click output settings. They also bypass Rich's formatting, creating a visually inconsistent experience.

**Recommended Remediation:**
Replace `print()` with `click.echo(..., err=True)` or pass a Rich `Console` instance.

**Verification:**
- Manual test: `gd-tools format` on a file with syntax errors → error appears in the same style as other commands.

#### M4. No file I/O or encoding error handling in lint/format

**Location:**
- `src/gd_tools/lint_runner.py` — file read in `run_lint()`
- `src/gd_tools/format_runner.py` — file read/write in `run_format()`

**Description:**
Neither `run_lint()` nor `run_format()` wraps file I/O in try/except. If a `.gd` file is encoded in a non-UTF-8 encoding, or if there are permission issues, the entire lint or format run crashes with an unhandled `UnicodeDecodeError` or `PermissionError`.

**Impact:**
A single unreadable file (e.g., a binary file with a `.gd` extension, or a file locked by another process) crashes the entire lint/format run with no useful error message.

**Recommended Remediation:**
Wrap file reads with `encoding="utf-8"` and try/except for `UnicodeDecodeError` and `PermissionError`. Log a warning and skip the file.

**Verification:**
- Unit test: `run_lint()` with a non-UTF-8 file → warning logged, other files still linted.

#### M5. Substring matching for plugin idempotency in init

**Location:** `src/gd_tools/init.py` — `enable_gut_plugin()`

**Description:**
The idempotency check for the GUT plugin entry uses `gut_entry in content`, which checks if the string appears anywhere in `project.godot` — including in comments, autoload names, or other unrelated sections. This can produce false positives (believing GUT is already enabled when it's only mentioned in a comment).

**Impact:**
`gd-tools init` may skip enabling the GUT plugin if the string `gut` appears anywhere in `project.godot`, even if the plugin is not actually enabled in `[editor_plugins]`.

**Recommended Remediation:**
Parse the `[editor_plugins]` section specifically (using `configparser` or a targeted section check) rather than substring matching the entire file.

**Verification:**
- Unit test: `project.godot` with `# gut is great` in a comment but no enabled plugin → `enable_gut_plugin()` adds the plugin.

#### M6. `project.godot` I/O without explicit UTF-8 encoding

**Location:** `src/gd_tools/init.py` — multiple `read_text()`/`write_text()` calls

**Description:**
`project.godot` is read and written using `read_text()` and `write_text()` without specifying `encoding="utf-8"`. On Windows, the default encoding may be `cp1252` or another locale-specific encoding. Godot expects `project.godot` to be UTF-8 encoded.

**Impact:**
On systems where the default encoding is not UTF-8 (common on some Windows configurations), `gd-tools init` may corrupt `project.godot` by rewriting it in a non-UTF-8 encoding, or may fail to read non-ASCII characters in the file.

**Recommended Remediation:**
Add `encoding="utf-8"` to all `read_text()` and `write_text()` calls that touch `project.godot`.

**Verification:**
- Unit test: on a system with non-UTF-8 default encoding, `project.godot` with non-ASCII characters is preserved correctly after init.

#### M7. LCOV report uses `res://` paths in SF records

**Location:** `src/gd_tools/coverage/lcov_reporter.py` — `SF:` record generation

**Description:**
The LCOV reporter uses Godot's `res://` protocol paths in `SF:` (source file) records. The LCOV format expects filesystem paths. Tools like `genhtml` cannot open `res://` paths and will fail to annotate source code.

**Impact:**
`genhtml` and other LCOV-consuming tools cannot generate browsable HTML reports from gd-tools LCOV output. They will show coverage numbers but no source code annotation, or may error out.

**Recommended Remediation:**
Convert `res://` paths to filesystem paths using the project root:
```python
fs_path = res_path.replace("res://", str(project_root) + "/")
```

**Verification:**
- Unit test: LCOV output `SF:` records contain filesystem paths, not `res://` paths.
- Manual test: `genhtml coverage.info` succeeds and shows source annotations.
